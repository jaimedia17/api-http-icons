# Authorization Token Icon  
**Jai Visuals NZ — API Security & Credential Objects**

## Overview  
The Authorization Token icon represents the **signed credential** a system uses to determine whether a request has permission to access a resource.  
In modern API workflows, tokens carry structured claims such as issuer, scope, audience, and expiry, allowing services to validate access without exposing user credentials.

This icon abstracts the idea of a **compact, self‑contained credential** used across distributed systems.  
It applies to OAuth, JWT, service‑to‑service authentication, IAM workflows, and backend permission checks.

## Geometry & Semantic Structure  
- **Capsule Boundary — Portable Credential**  
  Represents a sealed, transportable token that can be passed between services.

- **Circular Node — Signature / Cryptographic Marker**  
  Symbolizes the token’s integrity and verifiability.

- **Horizontal Lines — Structured Claims**  
  Abstract fields such as issuer, scope, expiry, and permissions.

- **Neutral, Minimal Form**  
  Avoids referencing any specific technology (JWT, OAuth, SAML, etc.).  
  Focuses on the semantic meaning of a token as a **signed, structured credential**.

## Authorization Domain  
This icon represents the **authorization phase** of an API workflow.  
It is used to illustrate:

- OAuth access tokens  
- JWT‑based authorization  
- Service‑to‑service authentication  
- Backend permission checks  
- IAM workflows  
- API gateway security  
- Microservice access control  

It fits naturally into:

- architecture diagrams  
- developer documentation  
- IAM system maps  
- API security flows  
- enterprise authentication diagrams  

The icon communicates **trust, verification, and controlled access**.

## Narrative Fit  
The icon aligns with the real behavior of authorization tokens:

1. A client or service obtains a signed token  
2. The token carries claims defining what the requester can access  
3. The token is attached to a request  
4. The system validates the signature and claims  
5. Access is granted or denied based on the token  

This abstraction works across all modern identity and access management systems.
