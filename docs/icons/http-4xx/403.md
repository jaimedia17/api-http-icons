# 403 — Forbidden Icon  
Authorization Failure at Permission Boundary

## Overview  
The **403 — Forbidden** icon represents a request that is understood, structurally valid, and authenticated (if credentials were provided), but explicitly denied due to insufficient permissions.  
Unlike a 401, the requester’s identity may be known — the failure occurs at the **authorization layer**, not the authentication layer.  
The server evaluates the request but refuses access to the protected resource.

## Geometric Meaning  
- **Left Circle — Request Entity**  
  The request is valid and recognized.  
  The server can interpret it and, if applicable, identify the requester.

- **First Dash — Authorization Boundary**  
  This is the permission checkpoint.  
  The request reaches this boundary but is not permitted to cross it.

- **Right Segment — Protected Resource Layer**  
  The resource exists and is reachable in principle, but the requester lacks the required permissions to access it.

## Forbidden Access Indicator  
- **“X” on the Authorization Boundary**  
  The failure is tied to **permissions**, not identity.  
  The rejection occurs *at the boundary*, not inside the request.  
  This placement communicates that the requester may be authenticated, but the action is not allowed.

## Narrative Fit  
The icon mirrors the technical sequence of a 403 response:

1. The request arrives and is structurally valid.  
2. The server may authenticate the requester (if credentials are provided).  
3. The server evaluates permissions.  
4. Authorization fails — the requester is not allowed to perform the action.  
5. The resource layer is reachable but intentionally withheld.  
6. The server returns **403 Forbidden**.

This abstraction aligns with the true meaning of 403 in API and HTTP workflows:  
**the requester is known but not permitted to access the resource or perform the operation.**
