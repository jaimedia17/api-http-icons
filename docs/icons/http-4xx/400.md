# 400 — Bad Request - Icon
Client Input Validation Failure

## Overview  
The **400 — Bad Request** icon represents a client‑side input failure where the server cannot process the request due to malformed, incomplete, or structurally invalid data.  
This icon is used in API documentation, backend dashboards, integration diagrams, and workflow states where the failure originates from **incorrect client input**, not server logic.

## Geometric Meaning  
- **Base Shape:** Square frame  
  - Represents a request envelope or payload boundary.  
- **Inner Slash / Break:**  
  - Indicates structural invalidity or malformed content.  
  - Communicates “cannot parse,” “cannot validate,” or “payload rejected.”  
- **Symmetry:**  
  - Maintains visual consistency with the 4xx family, ensuring recognizability at small sizes.

## Semantic Role  
This icon communicates:  
- malformed JSON  
- invalid query parameters  
- missing required fields  
- incorrect data types  
- schema validation failure  
- corrupted or truncated payloads  
- client‑side formatting errors

It should **not** be used for:  
- authentication failures (use **401**)  
- permission issues (use **403**)  
- resource absence (use **404**)  
- conflict states (use **409**)  
- semantic validation failures (use **422**)  
- rate‑limit enforcement (use **429**)

## Narrative Fit  
Use this icon when the narrative describes:  
- “The server could not understand the request.”  
- “The payload is malformed.”  
- “The request body does not match the expected schema.”  
- “Client input failed validation before processing.”

This icon signals **client responsibility**, not server error.

## Usage Examples  
- API reference pages showing validation rules  
- Error states in developer dashboards  
- Integration diagrams illustrating request rejection  
- Form validation flows  
- Webhook payload inspection tools
